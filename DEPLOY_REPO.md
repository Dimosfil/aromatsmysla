# Deploy Repository

Target git repository root:

```text
<this bot profile repository root>
```

Copy these files and folders to that repository:

- `.dockerignore`
- `.env.example`
- `.gitignore`
- `Dockerfile`
- `README.md`
- `DEPLOY_REPO.md`
- `content.seed.json`
- `env.bothost.example`
- `standalone/`

Do not copy or commit:

- `.env`
- `.env.local`
- real Telegram tokens
- real admin passwords
- runtime `data/`
- SQLite files
- logs
- generated `node_modules/` or `dist/`

On the hosting platform, use the deployment repository root, enable the custom
Dockerfile option, choose `Node.js`, enable a domain for the admin panel, and
upload variables from `env.bothost.example`. Leave the main file empty when the
custom Dockerfile option is enabled. Keep bot copy, channel button URL, guide
titles, button prefixes, media, and PDF paths out of env; first deploy reads
them from `standalone/bot/content.seed.json` in Docker layout or
`content.seed.json` in a flattened BotHost workspace layout, and later admin
edits are stored at `GUIDE_BOT_CONTENT_PATH`.

## BotHost Form Values

Use these exact values when creating the bot:

- Platform: `Telegram`.
- Development language: `Node.js`.
- Python version: ignore this field; this is not a Python project.
- Deployment location: `Netherlands`.
- Bot Token: paste the real BotFather token into the BotHost field.
- Git URL: `https://github.com/Dimosfil/aromatsmysla.git`.
- Branch: `main`.
- Use domain: enabled if the admin panel should be available from BotHost.
- Use custom Dockerfile: enabled.
- Main file: empty.

If BotHost shows or stores `standalone/packages/core/src/ports.ts` as the main
file, clear it. That file is a TypeScript type module, not an executable bot
entry point. A deploy log with `Unknown file extension ".ts"` means BotHost used
its autogenerated Dockerfile instead of this repository's `Dockerfile`.
